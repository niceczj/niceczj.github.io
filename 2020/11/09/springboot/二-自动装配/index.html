(二) 自动装配

发表于 2020-11-09 分类于 主流框架 ， Spring ， Spring Boot Valine：
本文字数： 12k 阅读时长 ≈ 11 分钟

Spring Boot 能够快速流行的最主要的原因，就是 Spring Boot 大大的降低了开发成本。
而能够快速开发的原理就是自动装配。

@SpringBootApplication

我们先观察下 Spring Boot 的入口类的@SpringBootApplication注解，是一个组合注解：

// 含有@Configuration，标记这是一个配置类
@SpringBootConfiguration
// 自动装配的核心
@EnableAutoConfiguration
// 扫描组件的标记
@ComponentScan(excludeFilters = { @Filter(type = FilterType.CUSTOM, classes = TypeExcludeFilter.class),
      @Filter(type = FilterType.CUSTOM, classes = AutoConfigurationExcludeFilter.class) })
public @interface SpringBootApplication {

除了配置类与扫描的注解，最重要的就是@EnableAutoConfiguration这个注解。

@EnableAutoConfiguration

@EnableAutoConfiguration的语义就是开启自动配置。

// 添加当前配置类的目录为自动配置目录
@AutoConfigurationPackage
@Import(AutoConfigurationImportSelector.class)
public @interface EnableAutoConfiguration

@Import

@Import用于导入配置类，支持以下三种类型：

  • 直接导入配置类（@Configuration类）。
  • 返回要导入的bean的名称数组（ImportSelector子类），通过selectImports方法注入Bean。
  • 自定义创建和注册bean（ ImportBeanDefinitionRegistrar子类），通过registerBeanDefinitions方法注入Bean。

@Import具体的实现是由ConfigurationClassPostProcessor处理（AnnotationConfigUtils#registerAnnotationConfigProcessors引入）。具体可以参考(三) ConfigurationClassPostProcessor。

条件注解

在自动配置类上几乎都有各种@ConditionOnXxx条件注解，比如：

@Configuration(proxyBeanMethods = false)
@ConditionalOnWebApplication(type = Type.SERVLET)
@ConditionalOnClass({ Servlet.class, DispatcherServlet.class, WebMvcConfigurer.class })
@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)
@AutoConfigureOrder(Ordered.HIGHEST_PRECEDENCE + 10)
@AutoConfigureAfter({ DispatcherServletAutoConfiguration.class, TaskExecutionAutoConfiguration.class,
      ValidationAutoConfiguration.class })
public class WebMvcAutoConfiguration

随便查看一个注解@ConditionalOnClass，可以看到是一个派生注解，是建立在@Conditional基础上的：

OnClassCondition的继承关系如下：

Condition

通过接口的注释可知，该接口是通过 Condition#match来决定 bean definition 是否被注册。

/**
 * A single {@code condition} that must be {@linkplain #matches matched} in order
 * for a component to be registered.
 *
 * <p>Conditions are checked immediately before the bean-definition is due to be
 * registered and are free to veto registration based on any criteria that can
 * be determined at that point.
 *
 * <p>Conditions must follow the same restrictions as {@link BeanFactoryPostProcessor}
 * and take care to never interact with bean instances. For more fine-grained control
 * of conditions that interact with {@code @Configuration} beans consider the
 * {@link ConfigurationCondition} interface.
 */
@FunctionalInterface
public interface Condition {
   boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata);
}

SpringBootCondition

SpringBootCondition的注释写明该类是 Spring boot 的@Condition实现的基类，提供日志方面的输出。

SpringBootCondition就是一个模板方法，留了一个getMatchOutcome钩子给子类去实现。

public abstract class SpringBootCondition implements Condition {

   @Override
   public final boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
      String classOrMethodName = getClassOrMethodName(metadata);
      try {
         ConditionOutcome outcome = getMatchOutcome(context, metadata);
         logOutcome(classOrMethodName, outcome);
         recordEvaluation(context, classOrMethodName, outcome);
         return outcome.isMatch();
      }
   }

   /**
    * Determine the outcome of the match along with suitable log output.
    * 确定匹配的结果和合适的日志输出，钩子方法，留给子类实现
    */
   public abstract ConditionOutcome getMatchOutcome(ConditionContext context, AnnotatedTypeMetadata metadata);
}

AutoConfigurationImportFilter

AutoConfigurationImportFilter可以过滤spring.factories中的自动配置类，如果同时实现了org.springframework.beans.factory.Aware Aware接口，AutoConfigurationImportFilter#match会在Awar#setXxx之后执行。

/**
 * Filter that can be registered in {@code spring.factories} to limit the
 * auto-configuration classes considered. This interface is designed to allow fast removal
 * of auto-configuration classes before their bytecode is even read.
 * <p>
 * An {@link AutoConfigurationImportFilter} may implement any of the following
 * {@link org.springframework.beans.factory.Aware Aware} interfaces, and their respective
 * methods will be called prior to {@link #match}:
 */
@FunctionalInterface
public interface AutoConfigurationImportFilter {

   /**
    * Apply the filter to the given auto-configuration class candidates.
    * 应用在
    */
   boolean[] match(String[] autoConfigurationClasses, AutoConfigurationMetadata autoConfigurationMetadata);

}

OnClassCondition

OnClassCondition是默认配置的AutoConfigurationImportFilter之一，我们从OnClassCondition的角度去看整个条件注释的流程。
在SpringBootCondition#match以及FilteringSpringBootCondition#match上打上断点，可以追踪到：

// ConfigurationClassParser.DeferredImportSelectorGroupingHandler#processGroupImports
public void processGroupImports() {
	for (ConfigurationClassParser.DeferredImportSelectorGrouping grouping : this.groupings.values()) {
		Predicate<String> exclusionFilter = grouping.getCandidateFilter();
    // grouping.getImports() 是 FilteringSpringBootCondition#match 入口
		grouping.getImports().forEach(entry -> {
			ConfigurationClass configurationClass = this.configurationClasses.get(entry.getMetadata());
			try {
        // processImports 是 SpringBootCondition#match 入口
				processImports(configurationClass, asSourceClass(configurationClass, exclusionFilter),
						Collections.singleton(asSourceClass(entry.getImportClassName(), exclusionFilter)),
						exclusionFilter, false);
			}
		});
	}
}

AutoConfigurationImportFilter 默认有三个配置：

ConfigurationClassFilter是AutoConfigurationImportSelector的一个内部类

// AutoConfigurationImportSelector.ConfigurationClassFilter
private static class ConfigurationClassFilter {
   // SPI 配置的 AutoConfigurationImportFilter
   private final List<AutoConfigurationImportFilter> filters;

   List<String> filter(List<String> configurations) {
      long startTime = System.nanoTime();
      String[] candidates = StringUtils.toStringArray(configurations);
      boolean skipped = false;
      for (AutoConfigurationImportFilter filter : this.filters) {
         // 调用AutoConfigurationImportFilter#filter
         // 实际上判断的核心是子类的getOutcomes
         boolean[] match = filter.match(candidates, this.autoConfigurationMetadata);
         for (int i = 0; i < match.length; i++) {
            if (!match[i]) {
               candidates[i] = null;
               skipped = true;
            }
         }
      }
      if (!skipped) {
         return configurations;
      }
      List<String> result = new ArrayList<>(candidates.length);
      for (String candidate : candidates) {
         if (candidate != null) {
            result.add(candidate);
         }
      }
      if (logger.isTraceEnabled()) {
         int numberFiltered = configurations.size() - result.size();
         logger.trace("Filtered " + numberFiltered + " auto configuration class in "
               + TimeUnit.NANOSECONDS.toMillis(System.nanoTime() - startTime) + " ms");
      }
      return result;
   }

}

OnClassCondition#getOutcomes也比较简单，就是获取类上面的注解值，然后Class.forName看是否能够实例化。
因为实例化比较耗时，OnClassCondition多起了一个线程并行执行。

@Order(Ordered.HIGHEST_PRECEDENCE)
class OnClassCondition extends FilteringSpringBootCondition {

   @Override
   protected final ConditionOutcome[] getOutcomes(String[] autoConfigurationClasses,
         AutoConfigurationMetadata autoConfigurationMetadata) {
      // Split the work and perform half in a background thread if more than one
      // processor is available. Using a single additional thread seems to offer the
      // best performance. More threads make things worse.
      // 如果有配置类和CPU有多个，拆分出一个线程背后执行
      if (autoConfigurationClasses.length > 1 && Runtime.getRuntime().availableProcessors() > 1) {
         return resolveOutcomesThreaded(autoConfigurationClasses, autoConfigurationMetadata);
      }
      else {
         OutcomesResolver outcomesResolver = new StandardOutcomesResolver(autoConfigurationClasses, 0,
               autoConfigurationClasses.length, autoConfigurationMetadata, getBeanClassLoader());
         return outcomesResolver.resolveOutcomes();
      }
   }

   private ConditionOutcome[] resolveOutcomesThreaded(String[] autoConfigurationClasses,
         AutoConfigurationMetadata autoConfigurationMetadata) {
      int split = autoConfigurationClasses.length / 2;
      // firstHalfResolver 是另起一个线程，执行resolveOutcomes会join插队。
      OutcomesResolver firstHalfResolver = createOutcomesResolver(autoConfigurationClasses, 0, split,
            autoConfigurationMetadata);
      OutcomesResolver secondHalfResolver = new StandardOutcomesResolver(autoConfigurationClasses, split,
            autoConfigurationClasses.length, autoConfigurationMetadata, getBeanClassLoader());
      // 从AutoConfigurationMetadata获取到配置类的@ConditionalOnClass value
      // 如果有，就 Class.forName 判断是否存在这个类
      ConditionOutcome[] secondHalf = secondHalfResolver.resolveOutcomes();
      ConditionOutcome[] firstHalf = firstHalfResolver.resolveOutcomes();
      ConditionOutcome[] outcomes = new ConditionOutcome[autoConfigurationClasses.length];
      System.arraycopy(firstHalf, 0, outcomes, 0, firstHalf.length);
      System.arraycopy(secondHalf, 0, outcomes, split, secondHalf.length);
      return outcomes;
   }
}

再看SpringBootCondition#match：

// ConditionEvaluator#shouldSkip
public boolean shouldSkip(@Nullable AnnotatedTypeMetadata metadata, @Nullable ConfigurationPhase phase) {

   for (Condition condition : conditions) {
      ConfigurationPhase requiredPhase = null;
      if (condition instanceof ConfigurationCondition) {
         requiredPhase = ((ConfigurationCondition) condition).getConfigurationPhase();
      }
     // condition.matches 会走 Condition#match接口
      if ((requiredPhase == null || requiredPhase == phase) && !condition.matches(this.context, metadata)) {
         return true;
      }
   }

   return false;
}

前面知道SpringBootCondition也是模板方法，直接看OnClassCondition#getMatchOutcome实现：

public ConditionOutcome getMatchOutcome(ConditionContext context, AnnotatedTypeMetadata metadata) {
   ClassLoader classLoader = context.getClassLoader();
   ConditionMessage matchMessage = ConditionMessage.empty();
   // 获取类中的@ConditionalOnClass 的值
   List<String> onClasses = getCandidates(metadata, ConditionalOnClass.class);
   if (onClasses != null) {
      // 还是  Class.forName 看能否实例化
      List<String> missing = filter(onClasses, ClassNameFilter.MISSING, classLoader);
      if (!missing.isEmpty()) {
         return ConditionOutcome.noMatch(ConditionMessage.forCondition(ConditionalOnClass.class)
               .didNotFind("required class", "required classes").items(Style.QUOTE, missing));
      }
      matchMessage = matchMessage.andCondition(ConditionalOnClass.class)
            .found("required class", "required classes")
            .items(Style.QUOTE, filter(onClasses, ClassNameFilter.PRESENT, classLoader));
   }
   List<String> onMissingClasses = getCandidates(metadata, ConditionalOnMissingClass.class);
   if (onMissingClasses != null) {
      List<String> present = filter(onMissingClasses, ClassNameFilter.PRESENT, classLoader);
      if (!present.isEmpty()) {
         return ConditionOutcome.noMatch(ConditionMessage.forCondition(ConditionalOnMissingClass.class)
               .found("unwanted class", "unwanted classes").items(Style.QUOTE, present));
      }
      matchMessage = matchMessage.andCondition(ConditionalOnMissingClass.class)
            .didNotFind("unwanted class", "unwanted classes")
            .items(Style.QUOTE, filter(onMissingClasses, ClassNameFilter.MISSING, classLoader));
   }
   return ConditionOutcome.match(matchMessage);
}

常见条件注释

  • 条件注解：@Conditional
  • Bean 条件：@ConditionalOnBean，@ConditionalOnMissingBean，@ConditionalOnSingleCandidate
  • 类条件：@ConditionalOnClass，@ConditionalOnMissingClass
  • Web 应⽤条件：@ConditionalOnWebApplication，@ConditionalOnNotWebApplication
  • 属性条件：@ConditionalOnProperty
  • 资源条件：@ConditionalOnResource
  • 其他条件：@ConditionalOnExpression，@ConditionalOnJava，@ConditionalOnJndi

总结

  2. 模板方法是一个非常常见的设计模式。
  3. 耗时功能可以拆分多个线程，但是线程的个数需要严格测试，不是越多越好。如：OnClassCondition#getOutcomes。
  4. 有个疑问，FilteringSpringBootCondition与SpringBootCondition功能有重叠，FilteringSpringBootCondition已经过滤一遍，为什么SpringBootCondition还要再次过滤？
